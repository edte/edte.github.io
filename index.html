项目经历

推送系统（2020.6.20 - 2021.5.15）

网校整个推送系统的实现，包括微信，短信等推送，我在这个项目负责后端开发。

在这个项目中，我遇到了这些问题：

首先，是不同推送方式格式的不同问题，如果不同推送设置不同接口会过于冗杂，故我又引入了推送模板的概念，在推送中心将所有信息加了一层，对外暴露相同的接口，使用模板 ID 调用下层具体推送。通过这个方式，提高了推送方式的扩展性，只需实现推送接口即可使用，同时增加了模板的概念，便于管理模板。

然后，是请求 JSON 的校验过于复杂问题，由于对外设计接口扩展性较好，故请求 JSON 的校验过于复杂。对于这个问题，我增加了校验层，使用了 JSON Schema 技术对其进行校验，然后分解调用下层具体的函数。通过这个方式，不但校验比较方便，而且易扩展，和其它成员合作也便于交流。

之后，我遇到了路由过于复杂，编写臃肿的问题，对于这个问题，我使用了表驱动模式，将数据和逻辑解耦，不但易读性提高，而且可扩展性也变强了。

然后，我又遇到了配置较多，编写时不便的问题，对于这个问题，我将不同配置内联到全局 conf 结构体中，然后使用了单例模式获取配置。使用这个方式后，在初始化、获取配置等功能上简介了许多。

同时，我遇到了需要对消息生命周期追踪的问题，我引入了每个消息的唯一 id，然后使用了 Wrapping Error 技术处理错误，每层加上层信息和错误往上抛，最后在顶层统一获取错误并处理日志。通过这个方式，不再需要逐层处理错误，并且能够很快定位到消息的生命周期，调试和获取消息信息速度大大增加。

 

 

十九届五中全会答题系统（2020.12.22 - 2021.1.21）

西南高校在线答题系统，我在这个项目中负责后端开发。

在这个项目中，我遇到最困难的问题是并发数和 QPS 较高， 使得响应时间 RT 较大。对于这个问题，我使用了 Redis 缓存降低了 MySQL 频繁的 IO，同时又引入了一个全局的 map 作为二级缓存，同时由于读写不均衡，又引入了读写锁降低使用互斥锁带来的消耗，然后又对 MySQL 加了一些索引，避免全表扫描。最后成功将 RT 从 5s 降到了 <100ms 级别。

然后，我还遇到了安全问题，有学生直接抓包进行发送请求。对于这个问题，我将 HTTP 换成了 HTTPS 协议，增加了 TOKEN 的检验中间件，然后对报文的 referer 进行了特判，同时引入 IP 白名单，将高频访问拉入白名单，最后对不合理成绩进行判断，再对报文进行 Base64 解密，最后成功将抓包问题降到了 10.

最后，我还遇到了需要设计排行榜的问题，我的解决办法是使用 Redis 的 zset 数据结构，利用跳表和内存访问避免查询时频繁 IO。

 

 

简易 STL 实现（2021.11.5 - 至今）

STL 库的 Go 简易实现，个人练手项目，实现了常见的容器和算法，保证了线程安全。

在编写过程中，我遇到了这些问题：

首先，是 Go 没有泛型无法编写通用容器和算法问题，我重新设计了迭代器，使用接口和迭代器模式解决了这个问题，同时使容器和算法达到完全解耦。

然后，我还遇到了 Go 没有重载，使得容器初始化参数函数臃肿问题，我使用了 Functional Options 模式解决了这个问题，相对于内联配置对象、使用 Builder 模式等方法，闭包和函数编程更加简洁和易扩展。

该项目目前实现了 STL 中的大部分算法和部分容器，测试覆盖率 13%.
